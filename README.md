# Git Lesson

## リモートリポジトリとローカルリポジトリとはそれぞれ何でしょうか？

・リモートリポジトリ
ネットワーク上やファイル上に存在する複数の利用者で共有するリポジトリの事。
リモートリポジトリが使用できるサービス（プラットフォーム）は「GitHub」「Bitbucket」「Backlog」等の代表的なものがあげられる。
・ローカルリポジトリ
ユーザーがローカルマシン上で作業するために利用するリポジトリの事。

## プッシュとマージの違いは何でしょうか？
プッシュはローカルからリモートへ編集内容を反映させ、
マージは、現在使用しているブランチに特定のブランチを取り込むこと。

## コミットとプッシュの違い
・コミット
ゲームでいうセーブのようなもの。
仮に何かのバグ等でデータが消えてしまっても、最後にコミットした場所からやり直すことができる。
・プッシュ
ローカルからリモートへ編集内容を反映させるもの。
コミットは作業にあたって、都度編集内容を保存していくもので、
その編集した内容をリモートへ反映させるかの違い。

## コミットのメッセージはどのように書いてあげるのが最適でしょうか？
チームで管理を行うことが多いので、
ログを見返したときに、誰がどこの指摘に対してどのような編集を行なったのかを
わかるようにすることが重要。
※各プロジェクトやチーム内でのルールも定められている部分もあるのでそれに従う。

3行で完結にメッセージを記述することわかりやすくコミット内容を伝えるコツ
1行目ーコミットのタイトルや概要
2行目ー空行
3行目ーコミット内容や詳細を記入する。


## ローカルでマージするフローと、プルリクエストでマージするフローの違いは何でしょうか？
レビューの依頼をするかしないかの違い。
コードレビューをせずに反映をしてしまうとどこか編集内容に不備があることに気づけないので、
プルリクエストをつくり、レビューをしてらってから問題がないようであればそのままマージすることができる。

・ローカルマージ
メリットーリモートブランチに更新するまでの過程を最小ステップで行えるためスピード感がある。
デメリットーローカルからリモートへそのまま編集内容をプッシュするので仮にもし不備内容があっても気づけない。

・プルリクエストマージ
メリットーリモートのmasterブランチにマージする前にレビュー担当に依頼をするのでもし編集内容に不備がないかのチェックをしてくれる。
デメリットーレビューチェックを行うためにプルリクエストをしないといけないので確実にミスなく作業を進めていく分、時間と手間がかかる。

## コンフリクトを起こしてしまった場合、どう対処すべきですか？
対処の方法は3つの編集パターンがある。
①先にマージされた変更内容を取り込む。
②後にマージしようとしている変更内容を取り込む。
③どちらの変更内容も取り込む。

コンフリクトとは。
（複数人がブランチを切って同じ箇所の編集を行なってマージした際に、
どの内容の記述を優先すればいいのかわからない場合に起こる現象。）
